[[wiki/codebase/files/packager|packager.py]] — confirms the live path down from `main.py`: `auto_packager.py`'s `EnhancedAgentSpawner` wraps this file's `AgentPackager`, one layer removed from `main.py` itself. [[wiki/codebase/files/auto_connect_system|auto_connect_system.py]] — **the most significant confirmed bug found this session**: the "auto-join all packaged agents on startup" feature cannot fire for anyone, ever, under the current code. `scan_agents_folder()` globs `DW_*`-prefixed directories; [[wiki/codebase/files/packager|packager.py]]'s actual current agent-directory naming is plain `{agent_id}`, no prefix — confirmed three independent ways (packager.py's own "no `DW_` prefix" comments at the real construction site, a matching comment in `main.py`, and the shared `Config.NPC_APPLICATIONS_DIR` root both files agree on). The scan runs twice on every server start and returns empty both times. Not an isolated typo: [[wiki/codebase/files/packager|packager.py]]'s own top-of-file docstring still describes the old `DW_{agent_id}` convention this glob was evidently written against — the naming changed in the actual construction code and three other references (this glob, and two comments inside packager.py itself) never caught up. Added to [[known-issues]] as a new, still-open item. A second, independent issue in the same unreachable code path: the `spawner` parameter threaded through `launch_all_agents()` down to `_launch_agent()` is never actually called — both the NPC and god branches construct `NPCAgent` directly, bypassing [[auto_packager]]'s `_post_spawn()` pipeline (breeding attachment, auto-packaging queue) entirely. Two small, non-corrective additions made to [[main]] (nothing there was wrong, just written before this file existed): the four `/api/autoconnect/*` routes `integrate_with_backend()` registers, and the `mark_connected()` call inside `/api/player_event`.

[[wiki/codebase/files/frontend_builder|frontend_builder.py]] — genuinely standalone (stdlib only, zero internal imports), called by [[wiki/codebase/files/packager|packager.py]] to build and copy the React frontend. Harmless (same value either way), but the stated intent and the reality don't match. Also notable: a named, specific recovery path for a real tooling regression (rolldown-vite v7 dropping its bundled esbuild minifier), not a generic retry-on-failure.

[[wiki/codebase/files/minecraft_launcher|minecraft_launcher.py]] — `UltimMCLauncher` (one UltimMC install: discover, copy, account, instance, mods, launch) and `MultiAgentLauncher` (one `UltimMCLauncher` per agent, on top). One thing worth recording plainly: an initial read of `launch_instance()`'s `-l` flag misread it as an instance selector, based on a cross-file pattern match against `packager.py`'s generated launcher — checked directly with Devlord, who's actually run this tool against this codebase, and confirmed it selects the Minecraft version here, not an instance or account. [[wiki/codebase/files/action_format_sync|action_format_sync.py]] — `ActionFormatter`, a testing/logging/replay utility converting between action arrays and the `controls` dict `communication_protocol.py` consumes. Its `GOD_ABILITY_NAMES` table is the exact file [[god-abilities]] already covers in real depth — cross-checked entry by entry against that page's existing table (all six gods, including the `summon_vexes`/`summon_fangs`/`retract_tentacles`/`emerge` additions) and confirmed to match exactly, no correction needed. One genuinely nice confirming find: this file's `god_to_controls()` carries its own independent copy of the same `DIM-03` ability-index-decoding fix already documented on [[god-abilities]] for `god_controls.py`'s `decode_action()` — the identical bug (treating a discrete integer output as a continuous range needing a mapping formula) existed in two separate files' parallel implementations, with matching fix comments in both, real evidence for how tightly these two files have to stay in lockstep rather than just an assertion of it.

[[wiki/codebase/files/agents_json_manager|agents_json_manager.py]] — 33 lines, a pure compatibility re-export of `mc_uuid.AgentNameManager` under an old name, nothing more. Confirms [[main]]'s already-documented `get_agents_manager` import resolves to this file's `get_manager()` exactly.

[[wiki/codebase/files/chat_backend|chat_backend.py]] — **resolves the thread flagged during [[main]]'s ingest**, and the answer is the opposite of what was speculated: not a third, possibly-current chat implementation, but the oldest and most disconnected one. A complete, standalone Flask + Flask-SocketIO server (the only non-FastAPI file in this whole ingest), confirmed orphaned by direct search — nothing anywhere imports it, its two push functions have zero callers. Its own `handle_user_message()` is an explicit stub (`# (Replace with planner/LLM later)`, hardcoded `"I heard: {text}"` replies, never touching `agent.brain`), despite the file's own header comment claiming `(finalized)`. Not dismissed as a forgotten scratch file, though: `flask`/`flask-socketio` are still listed in `requirements.txt` with purpose comments describing exactly this file's role, meaning this was a real, deliberate early direction for "the web interface" that got superseded by the FastAPI approach everything else uses, with the dependency lines simply never pruned. Also runs on port 8765 — the same port [[main]]'s dead `start_chat_interface` and [[chat_launcher]]'s live one both use for serving the built frontend — not a live collision since nothing calls this file automatically, but a real one if anyone ever ran it manually alongside the frontend server.

[[wiki/codebase/files/dw_controller|dw_controller.py]] — `ControllerRuntime`: real host-hardware access (OpenCV camera, `sounddevice` microphone), permission-gated, feeding directly into agent memory and emotion. Resolves both pointers carried forward as open threads: [[agent]]'s `_get_controller()` singleton and its four `/api/controller/*` routes, confirmed matching this file's constructor exactly, including a deliberate `__new__`-based special case for device detection without a full agent. Mostly confirmatory rather than a new discovery — one small, low-stakes note: `/api/controller/status`'s `hasattr()` guards check for attributes `ControllerRuntime.__init__` always sets unconditionally, the same minor defensive-against-a-guaranteed-interface pattern already documented at length for [[chat_system]], here harmless and small.

[[wiki/codebase/files/mc_uuid|mc_uuid.py]] — the most-referenced still-unread dependency across the whole session, confirmed to be exactly what every prior citation said it was: `AgentNameManager`, the single source of truth for `agents.json`. **New, still-open finding, added to [[known-issues]]**: this file's `get_minecraft_uuid()` and [[minecraft_launcher]]'s own `_offline_uuid()` both claim to compute Minecraft's offline-mode account UUID and don't agree — verified directly by comparing what each function actually hashes, not assumed from similar names. This file matches Java's well-documented `UUID.nameUUIDFromBytes()` exactly (no namespace prepending); `minecraft_launcher.py`'s `uuid.uuid3()` call does prepend a namespace UUID's bytes per RFC 4122, producing a different hash input and therefore a different UUID for the same username. Unlike the earlier `-l` flag question this session, both implementations are directly in this codebase and the external behavior they're each matching (or failing to match) is unambiguous, standard, documented JDK behavior — not a possibly-divergent external tool fork. Noted on [[minecraft_launcher]] directly.

[[wiki/codebase/files/validation|validation.py]] — six well-built Pydantic request-validation models (chat, file upload, agent spawn, controller activation, task switch, player event), closing out the Python-side ingest on a fitting note: **none of the six is used anywhere.** Confirmed by direct search across `main.py` and `agent.py`, the only two files defining routes these models are shaped for — zero references in either. The only mention anywhere outside this file is a PyInstaller hidden-import string in `config.py`, not a real usage. Every real route this ingest read validates its own input manually instead, ad hoc, per field, at the call site — a real, working, if less rigorous, alternative to the centralized validation layer sitting unused beside it.

**This closes the Python-side per-file ingest** — every file from `main.py` through `utils/validation.py` now has its own page. The Java mod layer and the Electron frontend remain topic-level only.

### Electron frontend — per-file pages (2026-07-05)

[[electron-frontend]] already covered this system well at the topic level from an earlier pass; this batch gives all nine of its source files (`dw_agent/electron/react-app/src/`) their own pages, the first non-Python per-file pages in this wiki. New naming note, since this is the first time it's come up: JSX per-file pages use the component's real PascalCase name ([[wiki/codebase/files/App|App.jsx]], [[wiki/codebase/files/ControllerSafety|ControllerSafety.jsx]], etc.), and the one collision this created — `main.jsx` vs. the already-existing `main.py` → `main.md` — is resolved as [[wiki/codebase/files/main_jsx|main.jsx]], underscore, matching the vault's existing per-file-vs-topic-level underscore/hyphen distinction rather than inventing a new one.

Two real corrections landed on [[electron-frontend]] itself. First: that page said `ControllerSafety.jsx` "gets its own page: [[human-controller-debug]]." Wrong — that page is a different, unrelated system, and its own text already warns against exactly this mix-up. `ControllerSafety.jsx` now has a real page, [[wiki/codebase/files/ControllerSafety|ControllerSafety.jsx]]. Second, more substantial: [[electron-frontend]] had flagged `agent.py`'s `broadcast_world_model()` as "wasn't traced this pass." Traced now — zero callers anywhere in the repository, and the same is true of `broadcast_activity()`. Combined with how `App.jsx` mounts [[wiki/codebase/files/WorldModelVisualizer|WorldModelVisualizer.jsx]] (only when `worldModelData` is truthy, which only that dead message type ever sets), the World Model panel doesn't just show stale data — it never appears in the running UI at all. The Activity Log fares slightly better: it can't receive backend-driven `web`/`controller` entries, but `App.jsx`'s own file-upload handler pushes its own entries client-side, so the panel isn't fully inert, just far sparser than its own icon-coding (four activity types) suggests it should be.

Smaller, self-contained findings from the individual files: [[wiki/codebase/files/MentalMatrixSimulator|MentalMatrixSimulator.jsx]]'s already-known disconnection from the backend (confirmed by Devlord, and independently) turned out to include a second unused prop (`agentId`, not just `onSimulationEvent`) once read in full. [[wiki/codebase/files/WorldModelVisualizer|WorldModelVisualizer.jsx]] has two more small gaps beyond its already-documented dead `onPredictionRequest` prop: a `showPredictions` toggle that's watched by the redraw effect but never changes what's drawn, and a `hoveredNode` state with real highlight-render logic behind it that nothing ever sets, since only click handlers exist, not mouse-move ones. [[wiki/codebase/files/ControllerSafety|ControllerSafety.jsx]]'s already-known hardcoded `BACKEND_URL`/`agent_id: "demo"` turned out to be _two_ separate hardcoded `BACKEND_URL` declarations (one per component defined in the file) and _four_, not three, literal `"demo"` occurrences — and, more importantly, a confirmed practical consequence rather than just a style note: `App.jsx`'s correctly-derived `agentId` isn't passed down to this component as a prop at all, so Controller Mode will target the wrong agent entirely for any agent not literally named `demo`. One clean cross-stack confirmation, no discrepancy found: `ControllerActivityMonitor`'s displayed stats fields match [[dw_controller]]'s `ControllerRuntime.get_stats()` exactly, field for field. And `App.jsx`'s own `WebAccessManager` sub-component has a small, real gap of its own: its backend-sync effect has an early-return guard on an empty website list, so removing the last allowed site never actually clears it on the backend side.

Confirmed, not corrected: this whole system genuinely isn't running as a packaged Electron desktop app despite the folder name and the `electron` npm dependency — no main-process file exists anywhere in the tree, and `package.json` has no `electron` launch script, only Vite's own `dev`/`build`/`preview`. [[electron-frontend]] already said this from an earlier pass; this batch independently re-confirmed it via `package.json` directly rather than assuming the earlier finding still held.
