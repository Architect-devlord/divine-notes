## Codebase — files/, expanded scope

Devlord flagged that the file list above was missing real files — a full re-scan of every `.py` file in `divine-world-core` found ~25 more, mostly foundational, heavily-cross-referenced-but-never-directly-read files. Covered this pass:

- [[wiki/codebase/files/config-two-copies|config-two-copies]] — **confirms and explains `CLAUDE.md`'s own long-standing "known open item"**: `AUDIT_REPORT.md`'s duplicate-Config-class finding. Not a stale copy — `py_backend/config.py` and `py_backend/ai_core/config.py` serve two genuinely different runtime contexts (repo-root-invoked vs. packaged-agent-invoked), plus one real, verified inconsistency in the `agent_spawner` module-exclusion list that matches neither copy's own comment. `CLAUDE.md`'s known-open-items list updated accordingly, and a new Conventions note added for the naming-collision-exception pattern this page itself uses.
- [[wiki/codebase/files/emotion|emotion.py]], [[wiki/codebase/files/personality|personality.py]] — the two small trait/state classes nearly everything else reads from. `emotion.py` surfaced a real mismatch: [[wiki/codebase/files/obs_builder|obs_builder.py]]'s `EMOTION_KEYS` has two entries (`curiosity`, `frustration`) `EmotionSystem` doesn't produce — two permanently-dead observation dimensions, not previously flagged anywhere.
- [[wiki/codebase/files/ai_core-init|ai_core-init]] (`ai_core/__init__.py`) — the package entry point; confirms which `config.py` copy is live in a packaged-agent context.
- [[wiki/codebase/files/logger_setup|logger_setup.py]], [[wiki/codebase/files/los_filter|los_filter.py]] — both small; the latter's mechanical catalog was missing even though its concept was already folded into [[observation-space]]'s narrative.
- [[wiki/codebase/files/brain_capsule|brain_capsule.py]] — the actual `BrainCapsule` class [[memory-and-braincapsule]] describes narratively.
- [[wiki/codebase/files/obs_builder|obs_builder.py]] — the canonical 128-dim observation layout every neural component consumes; see the `emotion.py` finding above.
- [[wiki/codebase/files/policy|policy.py]] (`rl/policy.py`), [[wiki/codebase/files/env|env.py]] (`rl/env.py`) — `TransformerPolicy`/`GodTransformerPolicy` (referenced constantly throughout this whole ingest, never directly read until now), and a simulated (not live-Minecraft) Gymnasium training environment whose class docstring still claims the pre-fix `Box(50,)`/`Box(11,)` shapes despite the constructor correctly using 128/13 — a fourth instance of the same stale-dimension pattern found three times already, one step short of an actual bug this time.
- [[wiki/codebase/files/agent_spawner|agent_spawner.py]] — where `NPCAgent` instances and their real Minecraft client processes actually get created; `GOD_CONFIGS` turns out to be where each god's default memory allocation and starting personality traits are actually defined.

**Self-caught correction, 2026-07-05**: the earlier [[wiki/codebase/files/personality|personality.py]] page was substantially wrong — it described a `describe()` method and an `adjust(trait, delta)` method that don't exist in the actual file, and missed `gender`/`GenderType`/`to_dict()`/`from_dict()`/`apply_update()`/`similarity()` and four module-level gender/breeding functions entirely. Found while chasing down a `from ai_core.mc_gender import ...` claim in `agent_spawner.py` that also turned out to be wrong (the real import is from `personality.py`). Both pages corrected; see `log.md`'s matching entry for the full account, including the likely cause (a batched file read that produced a plausible-sounding rather than carefully-verified description).

[[wiki/codebase/files/rl-init|rl/**init**.py]], [[wiki/codebase/files/breeding_system|breeding_system.py]] — the latter confirms the corrected `personality.py` connection directly (`can_breed`/`determine_child_gender`/`assign_npc_gender`/`GenderType` are all real imports there) and adds the exact god-role-flip logic [[breeding-system]] only described in prose, plus the full pregnancy/growth/trait-inheritance mechanics.

[[wiki/codebase/files/god_controls|god_controls.py]] — partially resolves [[god-abilities]]'s flagged open question about `transform`/`revert`: they _are_ selectable via the same `ability_idx` mechanism as every other ability (confirmed — `ability_names()` includes them), narrowing the open part to whether the Java side correctly routes those two specific strings to `GodDisguiseHandler` rather than `ServerGodAbilityExecutor`. [[god-abilities]] updated to reflect the narrower question.

[[wiki/codebase/files/chat_system|chat_system.py]] — this file routes GUI/game WebSockets to `agent.brain` defensively, via `hasattr()` checks and fallbacks, as if `agent.brain` might not always implement `process_language_input`/`learn_from_file`/`should_speak`/`generate_speech`. Also runs its own independent autonomous-speech loop, separate from [[cognitive_loop]]'s. **Resolved by [[wiki/codebase/files/main|main.py]]'s ingest, below**: it's neither the live path nor a meaningful parallel one — see that entry for the full account.

[[wiki/codebase/files/main|main.py]] — resolves the [[wiki/codebase/files/chat_system|chat_system.py]] open question directly: `main.py` never imports `chat_system.py`, `cognitive_loop.py`, or `communication_protocol.py` at all (zero references, confirmed by search), and neither does `agent.py` or `auto_connect_system.py`. The only surviving reference to `chat_system.py` anywhere in the codebase is one `try/except`-guarded, functionally-inert call from [[cognitive_loop]]. Real routing is split three ways instead: `agent.py`'s own `/ws`/`/chat` for direct NPC chat, `main.py`'s `/api/player_event`/`/api/breeding/event` for world-state events from the Java mod, and `main.py`'s `/api/agents/chat_heard` for god proximity-chat specifically — except that last one calls an `AgentProcessManager.get_chat_queue()` method that is never defined anywhere in the repository, so it 500s on every call. Two more god-related routes, `/api/gods/ability` and `/api/gods/transform`, turned out to be unimplemented stubs that echo their input back without touching the real ability pipeline (see [[god-abilities]] for how abilities actually fire). Also surfaced: a `main.py` docstring claiming god agents run a separate `LLMOracleBrain`, not `BrainCore`/`LanguageIntelligence` — not chased down this pass, but a real candidate explanation for `chat_system.py`'s `hasattr()` defensiveness; and `py_backend/utils/chat_backend.py` (queued, unread) defines its own same-named `handle_user_message`, a naming coincidence worth a look when `utils/` comes up. Devlord's direction for chat delivery going forward — route it through [[cognitive_loop]]'s own gated decision cycle rather than an independent ungated loop — is recorded on both pages; `cognitive_loop.py`'s `_decide()` was confirmed this pass to already default to a "do nothing this cycle" outcome and already gate speech behind `language_desire` plus a cooldown, so the mechanism exists, it just needs to stay the only path.

[[wiki/codebase/files/web_browser|web_browser.py]] — Playwright-backed browser tool, confirmed to match [[perception-and-actuation]]'s existing narrative exactly: a pure tool, `brain_core.should_browse()` decides when, [[cognitive_loop]]'s `_execute_web_browsing()` drives it directly. The real find here wasn't in this file — it was in cross-checking [[wiki/codebase/files/agent|agent.py]]'s already-written page against it: [[wiki/codebase/files/agent|agent.py]]'s page said the browser routes pass through `agent.browser`; every actual call site, including in that same file, uses `agent.web_browser`. Corrected directly on that page, along with a missing `web_browser.py` entry in its Files Required. Also surfaced: two of `agent.py`'s eight `/browser/*` routes are broken (`GET /browser/screenshot` calls a nonexistent `screenshot_jpeg()`; `POST /browser/scroll` gates its logic behind a `hasattr(..., "_page")` check that's never true, so it silently no-ops while reporting success) — neither is a `web_browser.py` bug, both are documented on both pages. Two more small things: `BeautifulSoup`/`BS4_AVAILABLE` is imported but never used anywhere in the file (every extraction goes through Playwright's own `page.evaluate()`), and page caching has no TTL — a cached page can go stale indefinitely for the life of the process.

[[wiki/codebase/files/packager|packager.py]] — confirms the live path down from `main.py`: `auto_packager.py`'s `EnhancedAgentSpawner` directly wraps this file's `AgentPackager` (`self.packager = AgentPackager(...)`), and `main.py` instantiates `EnhancedAgentSpawner`, not this file directly. Also documents, for the first time anywhere in this wiki, the ~330-line `launcher.py` template this file generates per agent — the actual entry point every packaged agent runs, including its own `mode`-dispatch fix (chat → lightweight speech loop only, minecraft/autonomous → full `CognitiveLoop`) attributed to the same "Consolidate Duplicate Implementations plan, Step 5" that fixed the identical bug shape in [[wiki/codebase/files/agent|agent.py]]'s `run_standalone_agent()`. **Two small corrections made to [[known-issues]]'s existing packaging-bugs writeup**, both confirmed against current code rather than assumed stale or assumed still-accurate: the UltimMC-folder-copy fix it describes now happens in a different method than attributed (`_setup_ultimmc()`, not `_create_portable_package()` — the end result is the same, since both paths point at the same directory, but the mechanism moved), and the "preferred launch pattern" code snippet it quotes has one drifted flag (`-l agent_name` in the historical doc vs. `-l AGENT_ID` in current code).

**A real mistake made and caught in this first batch**: several early drafts linked a topic-level page's name (`agent-runtime`, `language-system`, `reward-and-learning-stack`) inside a Files Required/Used In list as if it were the actual dependency, when the real files (`agent.py`, `brain_language.py`, `skill_tracker.py`) don't share that name at all. All five pages have been corrected — see `CLAUDE.md`'s Conventions for the rule going forward, and `log.md`'s [2026-07-05] entry for the full account.
