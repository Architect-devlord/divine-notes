# brain_language.py

💡 **Role**: language learned entirely from scratch — no pretrained tokenizer, no pretrained embeddings, vocabulary and grammar both grown from the agent's own experience. [[language-system]] already covers the staged progression concept and the multimodal grounding idea; this page adds the mechanical catalog, the exact stage-promotion thresholds that page explicitly flagged as untraced, and — the biggest find this pass — a two-part bug in `process_input()` that meant the entire chat-training pipeline had never actually run in production until it was fixed.

## Imports

`time`, `re`, `json`, `logging`, `hashlib`, `collections` (`deque`, `defaultdict`), `dataclasses`, `typing`, `numpy`, `torch`, `torch.nn` — all external/stdlib. No internal project imports at module level; `sycophancy_weight` is imported lazily from `ai_core.reward_system` inside `_sycophancy_weight()` only (see [[reward_system]]).

## Classes

- **`MultimodalGroundingTransformer`** (`nn.Module`) — a small transformer (token embedding + learned positional embedding + `TransformerEncoder` stack) with two output heads: `word_logits` (next-token prediction) and `predicted_context` (predicts the 32-d context vector back from the token sequence — a self-supervised auxiliary loss forcing the model to actually encode context-relevant information in its hidden states, not just memorize token transitions). Context is injected once, as an extra prepended "token" (a linear projection of the context vector), not concatenated at every position.
- **`OnlineBPETokenizer`** — byte-level, no pretrained merges. `observe(text)` does one BPE merge step per call (finds the single most frequent adjacent-token pair across a rolling window of recent texts, merges it, assigns it the next vocab ID) rather than the usual batch-trained-once BPE — vocabulary grows one merge at a time, live, as the agent accumulates experience. `tokenize()`/`decode()`, `state_dict()`/`load_state_dict()` (the latter uses `eval()` to reconstruction tuple-shaped merge-pair keys from their JSON-stringified form — worth knowing since `ast.literal_eval()` would be the safer choice for the same result, though the practical risk is low since these state dicts are always self-generated, not externally supplied).
- **`Vocabulary`** — thin wrapper composing an `OnlineBPETokenizer`; `get_id()`/`get_token()`, `most_common_tokens()`, `show_merges()` for `get_language_progress()`'s reporting.
- **`ContextSchema`** — **not on the topic page.** Fixed 32-d layout: indices 0–3 vitals (health/hunger/saturation normalized), 4–8 emotions (joy/fear/anger/trust/surprise, read via `.snapshot()` if available), 9–13 personality (`as_array()` if available), 14–17 spatial (position, normalized by a fixed 1000-block scale), 18–21 modality flags (has-visual/has-audio/has-inventory/is-conversation), 22–31 open "overflow" slots for whatever `_enrich_context()` adds (memory novelty/recency/count, aggregated recalled-memory emotion, conversation length/activity, topic overlap) — `extract()` writes named overflow keys into fixed slots via a class-level name→index map, so the transformer sees a consistent layout regardless of which overflow keys happen to be present on a given call.
- **`ConversationBuffer`** — bounded turn history (`deque(maxlen=50)`) plus a 120-second inactivity window (`is_active()`). `_extract_topic_words()` (static) is a simple stopword-filtered, length-≥4 content-word extractor, shared by `_recall_relevant()`'s memory search and `_enrich_context()`'s topic-overlap calculation. `conversation_seed_tokens()` builds a generation seed from the last few turns.
- **`LanguageIntelligence`** — the class `add_language_to_brain()` attaches. See below for the method catalog; [[language-system]] covers the staged-progression narrative already.

No top-level classes beyond these five plus the transformer.

## `LanguageIntelligence` methods

- `__init__(agent_ref)` — builds the transformer, tokenizer, schema, conversation buffer, and the `_partner_visit_counts`/`_partner_last_seen` dicts (see the `state_dict()` fix below).
- `process_input(text, context, speaker_id)` _(the main entry point — see "The big finding" below)_ — 6 broad steps: recall relevant memory, enrich context, tokenize + append to training buffer, run `_train_step()` if enough buffered examples, generate a reply, store the exchange and (stage ≥ 1) attempt visual grounding via `_store_exchange_in_memory()`.
- `should_grpo_chat()` — gate for scheduling `_background_grpo_step()`: conversation must be active, not already mid-pass, cooldown elapsed, and a minimum turn-stride since the last pass (cooldown alone could still fire every reply in a fast exchange; the turn count is the harder floor).
- `_sycophancy_weight()` — thin wrapper around `reward_system.sycophancy_weight()`, so this file and [[reward_system]] compute the identical trait→weight formula from one shared function rather than two that could drift apart.
- `_background_grpo_step(user_text, enriched_ctx, cv)` _(async — the only async method in this file)_ — samples 4 candidate replies to the same exchange that already got a live reply, scores each via `epistemic_reward()` (self-referential — no external grader), and runs a GRPO-style update: baseline-subtracted, std-normalized advantages, weighted log-prob loss, backprop through `self.model`/`self.optimizer` (not a second optimizer — reuses the one already training on regular chat). Two of its own fixed bugs, both about matching `_train_step()`'s exact conventions rather than reinventing them: candidate tokens must be shifted (predict token _t+1_ from tokens up to _t_) — an earlier draft scored candidates against their own un-shifted tokens, collapsing the score to "how confident is the model in the input it was just given" rather than an actual generation likelihood; and `self.model()` returns a dict (`word_logits`/`predicted_context`), not a 2-tuple, which an earlier draft unpacked incorrectly. Entirely background — runs after the live reply is already returned, errors caught and logged, never allowed to affect the live reply path. Runs entirely separately from `self_supervised_trainer.py`'s `grpo_update()` (see [[self_supervised_trainer]]) — a genuinely different implementation for a genuinely different domain (text tokens vs. action vectors), not obviously a duplication to consolidate the way that file's own possible `communication_protocol.py` overlap might be.
- `_get_memory()` — safe lookup of `agent.memory`, falling back to `agent.brain.memory` if not found directly on the agent.
- `_recall_relevant(text)` — combines recency (`mem.recall()`) and topical relevance (`mem.search()` over up to 4 extracted topic words) into one deduplicated (by first-200-chars text) result list.
- `_enrich_context(context, recalled)` — see `ContextSchema` above; computes the actual overflow-slot values from recalled memories and current conversation state.
- `_store_exchange_in_memory(user_text, agent_text, recalled, context, speaker_id)` — **carries the `partner:{id}` tagging fix** referenced from [[memory]]'s `search_by_tag()`: this is the method that actually reaches `UnifiedMemoryStore.remember()` → `ScyllaMemoryBackend.save_event()`, real persistent storage, not just the capped in-RAM cache — the tag is what makes one specific partner's full history queryable straight from ScyllaDB later, without every chat message paying a synchronous DB round-trip on the live reply path. Also where visual grounding actually happens: if the agent has a `vision` attribute with a `latest_frame`, calls `vision.ground_token(token, best_word)` — the co-present moment (a word said while looking at something) that was previously infrastructure without a caller (see "second finding" below).
- `_train_step()` — needs ≥4 buffered examples; samples the last 8, pads to the batch's max token length, computes next-token cross-entropy plus a 0.5-weighted context-reconstruction MSE loss.
- `_agent_context()` — pulls whatever attributes the agent currently exposes (health/hunger/position/visual/audio/inventory/emotions/personality) into a plain dict, defaulting to neutral values for anything missing.
- `_ctx_vec(ctx)` — routes a context dict through `ContextSchema.extract()` into an actual tensor, attaching `agent.personality` first if not already present.
- `_update_stage()` — **the exact thresholds [[language-system]] flagged as untraced**: stage 1 (proto-language) at ≥5 experiences and ≥15 vocab tokens; stage 2 (linguistic) at ≥50/≥50; stage 3 (advanced) at ≥200/≥200. Both conditions required at each threshold, and stages only ever advance (`max(new, self.language_stage)`), never regress.
- `_generate(input_tokens, ctx_vec, context)` — returns `""` at stage 0 (pre-linguistic; the agent literally cannot produce text yet). Autoregressive sampling, temperature 0.8 at stage ≥2 (more confident) vs. 1.2 below that (more exploratory), capped at 10 tokens, stopping early on a PAD or END token.
- `should_speak()` — stage ≥1, cooldown-gated, then either a strong-emotion override (any emotion magnitude > 0.6) or a sociability-scaled random roll.
- `generate_speech(context)` — three seed strategies in priority order: active conversation → conversational seed tokens; no conversation but relevant memory found → seed from that memory's text; true cold start → seed from the most recent context-window entry. Stores the result in memory the same way a real exchange would.
- `learn_from_file(file_path, filetype)` / `_learn_text(path)` — plain-text/`.md` only; splits on `.` into up to 100 sentences, tokenizes and buffers each (building vocabulary via `vocab.observe()` as it goes), runs up to 10 extra `_train_step()` calls once the buffer is large enough, then checks for a stage promotion.
- `get_language_progress()` — stage name/number, vocab size, merge count and 5 most recent merges, experience/update counts, buffer sizes, 20 most common tokens.
- `state_dict()` / `load_state_dict(state)` — **two fixes recorded directly in the code, both worth knowing**: `vocabulary_size`/`pattern_count` keys exist specifically because a separate BrainCapsule sidecar JSON summary reads those exact key names and would otherwise always show 0; and `partner_visit_counts`/`partner_last_seen` were pure in-RAM state with _no save path at all_ before this fix — every agent restart silently reset every returning visitor back to "first visit," even though the underlying conversation history itself survived fine in ScyllaDB (via the `partner:` tag above). `load_state_dict()` also handles migrating a pre-BPE, word-level vocabulary format by replaying each old word through `observe()`.

## Module-level functions

- `_agreement_marker(text)` / `_contradicts(a, b)` — lexical-heuristic epistemic signals (agreement phrase matching; negation-marker-gated word-overlap for contradiction). The code's own comment is explicit that these are "the same epistemic status as the RND/ICM curiosity bonuses already used elsewhere" — cheap, imperfect, directionally useful, never treated as ground truth. References a design document, "Chat & Web GRPO plan §3.3/§6," by name — not confirmed to exist anywhere in this vault.
- `epistemic_reward(candidate, user_text, agent, sycophancy_weight)` — combines an agreement-vs-user score (down-weighted by `sycophancy_weight` — the whole reason this parameter exists, so agreeing with the user isn't free reward for a highly conscientious/low-agreeableness agent) with a self-consistency check against the agent's own recent memory (`_contradicts()` against up to 5 recent memory texts) — the same reward function `_background_grpo_step()` scores candidates with, and the mechanism behind [[reward-and-learning-stack]]'s `evidence_r` framing.
- `add_language_to_brain(brain_instance)` — constructs `LanguageIntelligence` and assigns five bound methods directly onto the `brain` object (`brain.language`, `process_language_input`, `should_speak`, `generate_speech`, `get_language_progress`) — the same attribute-assignment integration pattern [[continual_learner]] uses, not the deeper method-replacement monkeypatching [[wiki/codebase/files/vision|vision.py]] does.

## The big finding: `process_input()` had never actually run

Documented directly in the method's own fix comment, and worth stating plainly since it recontextualizes everything else in this file: two independent bugs meant the entire chat-training pipeline described above — conversation buffering, training-buffer accumulation, familiarity tracking, epistemic scoring, background GRPO — had never actually executed against real chat traffic before this fix landed.

1. `'timestamp': time.time()` was written as `'timestamp': time` in an early line of the method — since the module imports `time` (the module) rather than `from time import time` (the function), this assigned the module object itself into a dict rather than calling it, and the method crashed on this line on every single invocation, silently, however it was actually called.
2. Independent of bug 1: the real `/chat` HTTP route's handler, `process_chat()` in `agent.py` (see [[agent]]), never called `process_input()` at all — it called `generate_speech()` directly. So even accounting for bug 1, this method's call path from real user-facing chat simply didn't exist.

Both are fixed now (the typo corrected; `process_chat()` updated to call `process_input()`), but it's worth being accurate about what this means for anything downstream that assumes this pipeline has been running: familiarity tracking, the epistemic-scored background GRPO pass, and grounding-via-conversation are all newly-live behavior, not long-running, battle-tested mechanisms.

## Second finding: visual grounding had no caller until this same file provided one

`vision.py`'s `OnlineVisualVocabulary.ground_token()` (see [[wiki/codebase/files/vision|vision.py]]) existed as complete, working infrastructure with nothing calling it — visual clusters accumulated and got auto-named (`visual_7`, etc.) but never received a human word. `_store_exchange_in_memory()` is where that gap closes: the co-present moment (a topic word from the current exchange, the agent's current visual token) is available at exactly this call site, so it's used as a noisy, accumulative "correlation hypothesis" grounding signal, gated to stage ≥1 so it doesn't fire during pre-vocabulary babbling.

## Problems (faced by traditional AI systems / LLMs)

Any language model trained on pretrained embeddings and a fixed tokenizer starts with a vocabulary and set of concept boundaries chosen by whoever built the pretraining corpus, not by what the specific model's own deployment actually encounters — the opposite problem from [[wiki/codebase/files/vision|vision.py]]'s pretrained-CNN objection, applied to language instead of sight. Separately: grounding language in perception (connecting a word to what it refers to) is a famously hard, usually-supervised problem — most systems either skip it entirely or require explicit labeled image-caption pairs.

## Solutions

Growing both the tokenizer (one BPE merge at a time, live) and the vocabulary purely from the agent's own experience means the resulting "language" is genuinely shaped by what this specific agent has actually said and heard, with no external corpus setting the boundaries in advance. Grounding gets a low-supervision answer: rather than requiring labeled pairs, correlate whatever word is being used with whatever visual cluster is currently in view, repeatedly, and let the noise average out over many co-occurrences — the same distributional-semantics intuition ("a word is characterized by the company it keeps") applied to a visual cluster instead of a neighboring word.
